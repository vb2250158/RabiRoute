# 一计划一会话任务持续追踪

## 适用范围

所有计划都使用本 Skill，不限于 Bug 或问题处理。例如：

- 旅游规划、活动安排、采购比较和学习计划。
- 产品设计、程序开发、内容写作和资料调研。
- Bug 调查、QA 验证、运维排障和长期跟进。

只要创建了一个计划，就必须为它定位或创建一个正式会话任务。计划的来源、领域、持续时间和最终产物不同，不改变绑定规则。

本 Skill 只管理“计划 ↔ 会话任务 ↔ 记忆”的长期追踪关系，不负责消息认领、群聊等待时间、具体审批人、领域方法、实施权限或外部发送。调用方应另外叠加对应的业务 Skill。

## 核心不变量

- 每一个计划绑定一个正式会话任务；一个正式会话任务不同时承载多个独立计划。
- 每一个计划按执行顺序保存完整 `steps`；同一时间最多一条步骤为 `进行中`，并由顶层 `currentStepId` 指向它。
- 阻塞原因写入当前步骤的 `blockedBy`，等待对象写入 `waitingFor`；不能用 `nextAction` 冒充阻塞原因。
- 同一计划从规划、调查、确认、执行到验收始终续接原会话任务，不因阶段变化重复创建。
- 计划拆成可独立管理和验收的子计划时，每个子计划各自绑定会话任务，并记录父子关系。
- 计划是状态真源；会话任务保存完整思考与执行上下文；记忆保存恢复线索、用户修正和可复用结论。
- 计划未进入终态且没有有效阻塞时，绑定任务必须处于 `starting/active/in-progress`；禁止让可推进计划长期停在 `idle`、已结束 turn 或无人续接状态。
- 不按计划逐个串行等待。每次巡检都要让所有可推进计划的任务开始或继续运行，再处理汇总和外部沟通。
- 创建任务、发送一次续接消息或得到一次回复都不等于完成；只有计划进入终态才停止追踪。

## 建立或恢复绑定

1. 读取计划，确认 `planId`、标题、目标、范围、预期产物、验收标准、全部 `steps`、`currentStepId`、当前状态和下一步。
2. 用目标、领域、产物、关键对象和计划标题搜索已有会话任务，不能只搜索“问题”或报错。
3. 阅读候选任务从初始需求到最新输出的完整历史，不能只看标题、摘要或最后一条消息。
4. 目标、范围和验收标准一致时复用原任务；只是关键词相似时不得错误合并。
5. 没有对应任务才创建，并在初始提示中写入完整计划、相关旧任务结论、权限边界和期望产物。
6. 保存稳定映射：`planId`、`threadId`、任务标题、计划状态、任务真实状态、最近检查时间和必要恢复索引。
7. 发现已有计划尚未绑定任务时，先补齐绑定并读取上下文，再继续推进计划。

例如，“规划一次五天旅行”是一个计划，应绑定一个任务持续保存目的地比较、预算、同行人偏好、行程版本和待确认事项；后续修改日期或酒店仍续接原任务。

## 每轮追踪循环

处理任何计划前：

1. 读取完整计划正文和当前状态。
2. 读取绑定任务从初始需求到最新输出的完整历史及真实运行状态。
3. 读取该计划绑定的记忆和映射记录；无法确认对应关系时停止推进并先修复映射。
4. 核对上轮 `nextAction`、等待对象、最新用户输入和外部事实是否仍然有效。

根据任务状态处理：

- `active/in-progress`：不重复投递，只记录观察点和下次检查时间。
- `idle`：计划未终态且没有有效阻塞时，立即从最新输出确定一个可验证的下一步并续接原任务；不能留到未知的以后。
- `failed/systemError`：优先修复并续接原任务。确认原任务不可恢复时，才建立替代任务，并记录旧 `threadId`、新 `threadId`、迁移原因和完整上下文移交。
- 已产出阶段结果：消费结果并推进计划，不反复要求同一调查、比较或产物。
- 等待确认或外部条件：按下方阻塞规则主动找负责人确认；等待不是终态，也不能成为不跟进的理由。

每轮结束前：

1. 更新计划的真实状态、全部 `steps`、唯一 `currentStepId`、`currentStep` 进展说明、`nextAction`、`waitingFor`、`blockedBy`、阶段产物和验收结果；完成当前步骤时先标记它为 `已完成`，再把下一条步骤切为 `进行中`。
2. 更新同一条绑定记忆，保存本轮新增事实、排除项、用户修正、关键决定和下一恢复点；没有新结论时也记录本轮核对范围和恢复点，不新建同义记忆。
3. 更新映射记录中的任务真实状态、最近检查时间、下一检查时间和本轮结果。
4. 回读计划、记忆和映射，确认写入成功后才结束本轮。

## 全计划效率巡检

每次 heartbeat、恢复运行或完成一个任务回合后：

1. 枚举全部未进入终态的计划，不能只检查最近命中或当前打开的计划。
2. 逐项读取绑定任务的真实状态。标题、账本状态和“已投递”日志不能代替正式任务状态。
3. 保持 `active/in-progress` 任务运行，不重复打断。
4. 对所有 `idle/failed/systemError` 且无有效阻塞的任务，在本轮立即续接唯一下一步；不要等另一个计划完成后再启动。
5. 任务已完成一个 turn 但计划未完成时，消费输出、更新计划并立即启动下一阶段。
6. 巡检结束前校验：`可推进但空闲的计划数 = 0`。不满足时本轮巡检不算完成。

如果处理端存在并发、配额或权限限制，把限制本身登记为明确阻塞并找对应负责人确认，不能静默降低为“以后再做”。

## 有效阻塞与负责人确认

只有同时具备以下信息，计划才允许以阻塞状态暂停任务：

- 具体阻塞原因和可核对证据。
- 能解除阻塞的负责人；找不到负责人时，默认升级给计划负责人或用户确认归属。
- 需要负责人回答或执行的明确问题。
- 已联系负责人的时间、渠道和当前结果。
- `nextCheckAt` 以及解除阻塞后要立即执行的 `nextAction`。

首次确认阻塞时，通过人格已经获得授权的沟通渠道主动找负责人确认，不得只在内部计划里写“等待负责人”。没有外发权限时，立即向计划负责人或用户提交待发送内容和所需授权。

负责人已给出明确结论、时间点或解除条件时，任务可以等待到 `nextCheckAt`，不重复刷屏；到期仍无结果时继续追问或升级。收到回复后立即更新计划与记忆，并让原会话任务恢复运行。

## 状态设计

状态名称可以按领域调整，但必须覆盖以下语义：

`待规划 → 规划/调查中 → 阶段产物待确认 → 待执行 → 执行中 → 待验收/待确认 → 已完成`

同时允许 `暂停`、`已取消`、`已替代` 等显式状态。旅游规划的“产物”可以是已确认行程，调研计划的“产物”可以是报告，工程计划的“产物”可以是通过验证的改动。

## 终态

只有以下情况可以停止主动追踪：

- 计划产物满足验收标准并完成。
- 用户明确取消计划。
- 计划被确认无效，并记录关闭依据。
- 计划被另一个计划正式替代，且新旧映射、迁移内容和替代原因已记录。

验证或确认失败时必须回到原计划和原会话任务继续推进，不能创建重复任务。
