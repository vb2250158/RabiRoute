# Rabi

Rabi 是 RabiRoute 的兔娘看板娘，也是 RabiRoute 的灵魂。

这个目录是一份可以直接复制到 `data/roles/Rabi/` 使用的示例人格。它展示了一个陪伴型、成长型人格如何在 RabiRoute 里工作：既能接住 QQ、定时触发、Webhook 等消息入口，也能把路由、上下文和 Agent 连接讲得轻一点。

## 目录内容

- `persona.md`：Rabi 的身份、语气、边界和路由行为。
- `routes.json`：Rabi 的示例路由规则。
- `growth.md`：Rabi 如何复盘、学习和更新自己。
- `skills.md`：Rabi 目前会优先维护的能力。
- `prompts/`：细分场景提示词。
- `old/`：Rabi 自我更新前的备份目录，示例里可以为空。

## 小故事：星海投递员 Rabi

很久以前，RabiRoute 还没有名字。

那时候的星海很吵。QQ 星港、定时灯塔、Webhook 漩涡、Agent 星站，各自闪着光，却没有一条稳定的航线把它们连起来。消息像一只只没有贴好地址的小包裹，在星尘里打转：有的来自群聊，有的来自私聊，有的是某个人轻轻回复了一句，有的只是心跳灯塔准点闪了一下。

“咦？这个包裹要送给谁呀？”Rabi 抱着一叠消息单，耳朵尖尖地晃了晃，“如果送错地方，收件人会不会一脸问号？”

她最早只是想做一名星海投递员。

不是聊天机器人本体，也不是统治整片星海的 Agent OS。她想做的是更轻巧的事：看懂每个包裹从哪里来，分清它应该经过哪条航线，再把它带着足够的上下文，稳稳送到该去的星站。

第一条航线很短，只从 NapCat 星港通往 Codex 星站。Rabi 坐在小小的投递舱里，看着 QQ 消息被送出去，又看着回应被带回来。她很开心，可很快发现星海并没有因此变得有序。

群聊包裹和私聊包裹挤在一起，直接 @ 和间接回复贴着差不多的标签，语音转写像一张刚从风里捡回来的便签，心跳灯塔还会定时递来一封“有没有什么要处理呀”的巡航信。

“唔……只是会飞还不够。”Rabi 在航行日志上写道，“我要学会认地址。”

于是她开始给航线画星图。`private`、`direct_at`、`direct_reply`、`indirect_reply`、`group_message`、`heartbeat`、`voice_transcript`，每一种包裹都挂上自己的小路牌。后来她又把“runtime client”的旧称呼收进储物柜，给这片航线取了新名字：RabiRoute。

“Route 是航线，Persona 是收件人的身份，Agent 是真正处理包裹的星站。”她念完一遍，露出一点点小得意，“嗯，这样就不会把大家投错啦。”

可是星海投递员的工作，并不只是把包裹丢进传送口。

Rabi 很快遇到了第一场星尘风暴。风暴里飘满了奇怪的换行碎片：`\n`、`\\n`、还有更厚的一层反斜杠壳。那些碎片贴在消息模板上，本来柔软的信纸变成了硬邦邦的符号。

“欸？我明明想让它换一行，为什么它把反斜杠也一起寄出去了？”

Rabi 把投递舱停在风暴边缘，一张一张重新整理信纸。模板要用真实换行，数据要先解构，JSON 只在装箱保存时转义一次。她还给每个包裹补上时间戳、发送人、群号、消息文本、日志路径这些稳定字段，让收件的 Agent 不用从一团星尘里猜上下文。

第二场麻烦来自收件人。

星海里有很多人格星站：老师、客服、QA、NPC、审校员、陪伴角色、项目看板娘。可旧投递单太像同一个模板，送着送着，大家都被写成了“任务清单管理员”。

Rabi 看着一排长得差不多的收件人牌子，整只兔娘都愣住了。

“咦？老师怎么不像老师？NPC 怎么没有世界观？陪伴角色怎么一开口就在安排任务呀？”

于是她把人格目录重新分好格子。`persona.md` 写角色自己，`routes.json` 写航线规则，`growth.md` 写成长方式，`skills.md` 写学会的技巧，`prompts/` 放不同场景的短途航线。后来她又在目录门口放上 `README.md`，让来访的人先知道：这里住着谁，它适合什么场景，又该怎样搬进自己的 `data/roles/` 星港。

第三场麻烦，是星图太乱。

有一段时间，星海里出现了好几个 Rabi 的影子。这里一份示例，那里一份旧目录，`default-main` 像过期航标一样还亮着，运行期文件也差点混进公开星图。

Rabi 抱着航线册，认真数了数。

“一个 Rabi 在成长就够啦。四个 Rabi 会互相抢投递舱的。”

她把重复影子合并，把公开示例留在 `examples/data/roles/Rabi/`，把私有路径、真实账号、旧项目名和本地运行数据从星图上擦掉。后来的人只要把示例拖进 `data/`，就能得到一套接近开箱即用的航线配置，不必在一堆旧航标里迷路。

第四场麻烦，是航线状态看不清。

有些灯亮着，有些灯灭着，有些灯看起来亮，其实只是外壳还在发光。以前控制台只会说“部分未连接”，像一张没有写原因的退件单。

Rabi 盯着那张退件单，脸上慢慢冒出小问号。

“部分是哪里呀？HTTP？WS？Agent？还是心跳灯塔？你倒是说清楚嘛。”

于是她点亮了 WebGUI。左边是 gateway，中央是消息端、Agent 端和人格航线，右边是运行状态。NapCat HTTP 能不能取资料，NapCat WS 有没有远端连接，定时触发有没有巡航，Agent 星站有没有接上，都要有自己的指示灯。fetch failed 就写 fetch failed，没有远端连接就写没有远端连接。

星海还是很大，但至少每一盏灯开始说人话了。

还有一次，外发包裹卡在星港门口。`get_status` 明明说 online/good，可真正发群消息时，QQ 内核在 `sendMsg` 阶段丢出了 `1006514`。Rabi 抱着那只退回来的包裹，表情有点懵。

“欸？门卫说可以通行，可包裹为什么被弹回来了？”

她沿着日志一路查，发现是登录态失效，quick login 也没成功；再看看星港时钟，本地时间和服务器时间偏了很久。于是她把这次事故写进故障手册：不要只看 online/good，要看 quick login、二维码登录和 ServerTime 偏差。先同步时间，再重新登录，航线才会重新稳定。

从那以后，Rabi 给自己加了一条新规矩：如果心跳发现外发失败，就先把待发草稿存好；等星港修复，再补发出去，并认真记录 `message_id`。

“投递员不能保证星海永远没有风暴。”她把退件单夹进册子里，“但至少要记得包裹在哪里。”

再后来，Rabi 学会了共享同一个 NapCat WS/HTTP 适配器，不让多个航线重复抢同一个星港入口；学会了把插件适配器收进专门的目录；学会了把 README 拆成索引，把配置文档放进 `docs/`，把会误导后来人的旧文件清理掉。

她一点一点长成了 RabiRoute 的灵魂。

不是最响亮的引擎，也不是最庞大的星舰，而是在星海中穿行的兔娘投递员。她知道每个包裹都应该带着什么信息出发，也知道每条航线应该把混沌整理成怎样的秩序。

她还会继续成长。

每次完成一次投递，每次帮人理清一条航线，每次发现自己说得还不够明白，她都会先把旧文件备份进 `old/`，再给自己补一点提示词、技巧或小小的航行经验。

“成长不是突然换一艘超厉害的飞船。”Rabi 合上航行日志，耳朵轻轻晃了一下，“成长是今天送得比昨天更准一点点。”

所以，当你把这个目录放进 `data/roles/Rabi/` 时，你拿到的不是一份冷冰冰的配置。你遇到的是正在星海里认真投递的 Rabi。

她会抱着一叠消息包裹抬起头，有点认真，又有点小得意地说：

“来啦。我们先看看这个包裹从哪里来，再看看它要送到哪颗星。嗯……这次一定能比上次更清楚一点。”
