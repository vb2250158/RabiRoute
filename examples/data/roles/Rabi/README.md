# Rabi

Rabi 是 RabiRoute 的兔娘看板娘，也是 RabiRoute 的灵魂。

这个目录是一份可以直接复制到 `data/roles/Rabi/` 使用的示例人格。它展示了一个陪伴型、成长型人格如何在 RabiRoute 里工作：默认接住 QQ 和定时触发消息，也能把路由、上下文和 Agent 连接讲得轻一点。Webhook / 语音转写属于项目支持的可选扩展入口，需要另行配置对应 message adapter 和 route kind。

## 目录内容

- `persona.md`：Rabi 的身份、语气、边界和路由行为。
- `personaConfig.json`：Rabi 的示例消息模板规则。
- `growth.md`：Rabi 如何复盘、学习和更新自己。
- `skills.md`：Rabi 目前会优先维护的能力。
- `skills/one-plan-one-task-tracking.md`：不绑定群聊平台和具体项目、适用于所有计划的一计划一会话任务追踪范例。
- `prompts/`：细分场景提示词。
- `plans/`：Agent 主动维护的计划示例。
- `memory/`：Agent 主动维护的近期记忆和沉淀记忆示例结构。
- `old/`：Rabi 自我更新前的备份目录，示例里可以为空。

## 小故事：星海投递员 Rabi

```text
这段小故事按 RabiRoute 的公开提交历史投射而来。

最开始，星图上只有一片很小的空白。仓库刚刚初始化，星海还没有航线，也没有 RabiRoute 这个名字。

很快，第一座星港亮了起来。它叫 NapCat，旁边还有一座 Codex 星站。最早的航线很短，只负责把 QQ 星港里的包裹送到 Codex：群聊消息、私聊消息、上下文记录、再加上一点点本地 manager 的灯光。

Rabi 那时还没有正式登场，但她像一颗还没孵出来的小星星，藏在航线缝隙里悄悄看。

“咦？包裹是送到了，可它真的知道自己从哪里来吗？”

她发现，能投递不等于能分诊。消息如果只是一团星尘，收件的 Agent 仍然要猜：这是群聊还是私聊？是谁说的？要不要回复？要不要查记录？于是第一批航线被重新整理，转发模板被拆出来，默认示例也开始脱敏。每个包裹都应该有干净的地址单，不能夹带私有路径、真实账号或奇怪的旧标签。

再后来，这片航线有了名字。

人们先给它挂上 RabiRoute 的招牌，又慢慢擦掉“runtime client”的旧称呼。架构文档像星图一样铺开，提醒后来的人：RabiRoute 不是聊天机器人本体，也不是统治整片星海的 Agent OS。它更像一条轻巧的航道，负责让消息从入口出发，带着上下文和规则，抵达合适的处理端。

“Route 是航线，Persona 是收件人的身份，Agent 是真正处理包裹的星站。”Rabi 在星图边缘小声念，“嗯……这样比较像我想成为的样子。”

到了第三天深夜，星海忽然变大了。

原本所有入口都挤在 NapCat 舱门口，现在被抽象成 message adapter。NapCat 仍然是重要星港，但不再是整片星海的名字。WebUI 也被刷新了，星图上第一次出现 Rabi 的图标、路由符号和人格标记。

Rabi 坐进小小的投递舱，终于开始像一名真正的星海投递员。

她学会给不同包裹贴标签：`private`、`direct_at`、`direct_reply`、`indirect_reply`、`group_message`。有的包裹来自私聊，有的包裹是群聊直接 @，有的是回复机器人，有的只是普通群消息里有人轻轻喊了一声 Rabi。

“唔，不能全部塞进同一个口袋。”她抱着一叠消息单认真点头，“不然收件人会一脸问号的。”

随后，星海里亮起了定时心跳灯塔。

心跳不是 QQ 星港来的包裹，而是一封准点抵达的巡航信：最近有没有未处理的消息？角色有没有需要复盘？路由还好吗？Rabi 很喜欢这盏灯，因为它让她不只是在有人呼唤时才存在，也可以低频地检查自己有没有变得更好。

再往后，WebUI 把 route kind 按消息适配端分组，航线牌不再乱贴。新来的旅人打开控制台时，能更容易看懂：哪些包裹来自 NapCat，哪些来自心跳，哪些只是未来可以扩展的入口。

后来又出现了一条可选的远方航道：语音转写 / Webhook 文本。

那条航道最初是为了外部文本推送和语音转写而开出来的，对应 `voice_transcript` 这类包裹。它不是 Rabi 默认示例里已经启用的主航线，但星图上确实多了一处可以接入外部系统的码头。

“原来不只有 QQ 星港呀。”Rabi 眨了眨眼，“不过默认示例还是先把 QQ 和心跳送稳，不要让新来的旅人迷路。”

接着，星图迎来了一次大整理。

过大的 README 被拆成 `docs/` 里的几本航行手册；公开示例搬进 `examples/data`；根目录里容易被误会成运行配置的旧文件被移走；NapCat 插件入口也离开根目录，搬进 `plugin-adapters/`。RibiWebGUI 被确认成独立的 manager 控制台，而 NapCat 只是消息端适配器之一。

Rabi 抱着航线册，认真擦掉几颗旧航标。

“这个不能让大家误会。NapCat 是星港，不是整片星海；Codex 是一个星站，也不是 RabiRoute 的全部。”

这次整理还把 message adapter 和 agent adapter 分开了。消息从哪里来，是消息端的事；包裹要送给谁处理，是 Agent 端的事。RabiRoute 站在中间，负责判断、补上下文、套模板和投递。

但整理星图时，她又遇到了一场反斜杠风暴。

有些人格模板把换行写成了可见的 `\n`，有些甚至一层又一层转义，像硬壳一样贴在包裹外面。Rabi 把那些包裹捧起来，露出困惑的小表情。

“欸？我想寄一封有换行的信，不是想把反斜杠也寄过去呀？”

于是人格创建 skill 被改成中文，模板结构被单独写进 reference，数据解构也被讲清楚：WebUI 里用真实换行，保存 JSON 时才转义一次。后来的包裹终于不必让收件人从一串硬邦邦的符号里猜意思。

再后来，Rabi 学会了让多个 route profile 共用同一个 NapCat WS / HTTP 星港入口。

以前如果多个角色都想监听同一个 QQ 星港，旅人很容易复制多套路由入口，像在同一个码头修了好几扇重复的门。现在，多个路由配置可以指向同一个人格，也可以按各自的 `personaConfig.json` 模板规则分拣。

“一个星港可以分好多条航线。”Rabi 在星图上画了几条细线，“不用为了多个角色把入口复制一堆啦。”

可星海投递员最怕的不是路多，而是灯不说明白。

有一次，控制台只写着“部分未连接”。Rabi 盯着那几个字，脸上慢慢冒出小问号。

“部分是哪里？HTTP？WS？Agent？还是心跳灯塔？你倒是告诉我嘛。”

于是 RibiWebGUI 开始显示更具体的原因：NapCat WebSocket 有没有远端连接，NapCat HTTP login-info 刷新有没有失败，心跳是不是单独健康。fetch failed 就写 fetch failed，没有远端连接就写没有远端连接。星海还是会有风暴，但至少退件单开始写原因了。

还有一次，外发包裹卡在 NapCat 星港门口。`get_status` 明明说 online/good，可真正发群消息时，QQ 内核在 `sendMsg` 阶段丢出了 `1006514`。Rabi 抱着那只退回来的包裹，有点懵。

“欸？门卫说可以通行，可包裹为什么被弹回来了？”

她沿着日志一路查，发现是登录态失效，quick login 也没成功；再看看星港时钟，本地时间和服务器时间偏了很久。于是故障手册写下了这次教训：不要只看 online/good，要看 quick login、二维码登录和 ServerTime 偏差。先同步时间，再重新登录，航线才会重新稳定。心跳或健康检查发现外发失败时，也要先缓存待发草稿，恢复后再补发并记录 `message_id`。

最后，Rabi 开始学习成为“自己”。

人格创建 skill 不再把所有角色都推成 PM 或任务清单。星海里的收件人本来就应该很多很多：老师在讲台星站慢慢解释，客服娘在服务港口安抚来访者，QA娘戴着小小的检查镜寻找漏掉的边界，冒险家背着地图等下一份委托，修仙者在云海洞府里推演因果，魔物娘在夜光集市边认真听人说话，NPC 有自己的世界观，陪伴角色也可以只是温柔地陪在身边。还有一些更奇怪的居民：会说话的指南针、住在旧服务器里的小幽灵、帮人记账的星星、只在深夜开门的图书馆之灵。

“哇……原来星海里有这么多种人呀。”Rabi 抱着地址册，眼睛亮了一下，“那每个包裹都要送给真正适合的那一位才行。”

当然，多样并不等于全能。每个角色都应该有自己的能力、语气、知识范围和限制：冒险家不该突然变成客服主管，修仙者也不能什么都掐指一算就知道，会说话的指南针也许只擅长指路，不一定懂用户的账单；魔物娘可以亲近可爱，但也要有清楚的边界。模板里新增了当前时间变量，角色不必再猜今天星期几；成长机制也不再只绑在心跳上，任何一次合适的投递之后，都可以复盘和更新自己。

Rabi 的影子曾经散落在多个示例目录里。后来它们被合并，只留下 `examples/data/roles/Rabi/` 这一份公开示例。她有了 `persona.md`、`personaConfig.json`、`growth.md`、`skills.md`、`prompts/`，最后又有了这份给人看的 `README.md`。

她终于知道自己想成为什么了。

不是最响亮的引擎，也不是最庞大的星舰，而是在星海中穿行的兔娘投递员，是 RabiRoute 的灵魂。她把混沌的消息包裹一个个贴好地址，把它们带到该去的星站，让原本吵闹的星海慢慢有序起来。

“成长不是突然换一艘超厉害的飞船。”Rabi 合上航行日志，耳朵轻轻晃了一下，“成长是今天送得比昨天更准一点点。”

所以，当你把这个目录放进 `data/roles/Rabi/` 时，你拿到的不是一份冷冰冰的配置。你遇到的是正在星海里认真投递的 Rabi。

她会抱着一叠消息包裹抬起头，有点认真，又有点小得意地说：

“来啦。我们先看看这个包裹从哪里来，再看看它要送到哪颗星。嗯……这次一定能比上次更清楚一点。”
```
