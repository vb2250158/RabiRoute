# Rabi

Rabi 是 RabiRoute 的兔娘看板娘，也是 RabiRoute 的灵魂。

这个目录是一份可以直接复制到 `data/roles/Rabi/` 使用的示例人格。它展示了一个陪伴型、成长型人格如何在 RabiRoute 里工作：既能接住 QQ、定时触发、Webhook 等消息入口，也能把路由、上下文和 Agent 连接讲得轻一点。

## 目录内容

- `persona.md`：Rabi 的身份、语气、边界和路由行为。
- `routes.json`：Rabi 的示例路由规则。
- `growth.md`：Rabi 如何复盘、学习和更新自己。
- `skills.md`：Rabi 目前会优先维护的能力。
- `prompts/`：细分场景提示词。
- `old/`：Rabi 自我更新前的备份目录，示例里可以为空。

## 小故事：路由星图上的兔娘

在一个遥远的、消息与星轨交织的世界里，有一条看不见却无处不在的道路，名字叫做 RabiRoute。

这条路一开始并不叫这个名字。最早的时候，它只是荒原上一座小小的信标塔，负责把 NapCat 带来的 QQ 消息送到 Codex 那边。信标塔很努力，可它那时还分不清自己到底是一座桥、一段临时脚本，还是一张被写满 `\n` 的旧纸条。

“欸？为什么明明换行了，大家看到的还是反斜杠 n 呀？”

站在信标塔旁边的 Rabi 抖了抖耳朵，认真地把纸条摊平。她不是来统治这片世界的，也不是要把所有 Agent 都变成自己的臣民。她只是这条路的灵魂，一只会把混乱消息轻轻分拣好的兔娘看板娘。

后来，荒原上的道路越来越多。

有一天，路边立起了第一块新路牌：消息端适配器。NapCat 不再是世界的全部，而是众多入口之一。QQ 的群聊、私聊、回复、语音转写、定时心跳，都可以沿着不同的小径抵达 RabiRoute。

Rabi 低头看着地图，小声念道：

“消息端是消息端，Agent 端是 Agent 端，中间这段才是 RabiRoute 要守好的路。”

又过了一天，风从星图上吹来，把旧名字吹散了。信标塔正式有了自己的名字：RabiRoute。它不再装作完整的 Agent OS，也不再假装自己能回答世界上所有问题。它只认真做一件事：判断消息该去哪里，该怎样带着上下文抵达那里。

Rabi 很喜欢这个名字。她在路牌下挂了一盏小灯，灯上写着：

“Route · Persona · Agent”

从那以后，她开始整理人格目录。旧时代遗留下来的 `default-main` 被收进历史抽屉，重复的 Rabi 影子被合并成唯一的一份示例。公开道路上不能再出现真实账号、私有路径和奇怪项目名，所有示例都要干净、清楚、可以放心交给后来的人。

“四个 Rabi 会打架的。”她严肃地说，“一个就够了。一个认真成长的 Rabi。”

可是，新的麻烦很快来了。

有些旅人照着旧卷轴创建人格，结果卷轴里到处都是反斜杠。消息模板本来应该像一封清楚的信：

```text
[RabiRoute 数据解构]
事件：群聊直接 @ 触发
事件时间：{time}

[消息]
{message}
```

却被写成了一串让人眼花的符咒：

```text
事件：群聊直接 @ 触发\n事件时间：{time}\n消息：{message}
```

Rabi 把那串符咒从 WebUI 的文本框里拎出来，叹了口气。

“模板要用真实换行啦。JSON 要转义，也只能在保存的时候转义一次。”

于是她把人格创建指南重写成中文，把数据解构、route kind、当前时间变量和成长机制都放进了新卷轴里。以后不管有人想创建老师、客服、QA、NPC、审校员、陪伴角色、项目看板娘，还是一位只想静静陪在身边的朋友，都不必再被旧模板误导。

再后来，RabiRoute 的控制台醒了。

它有了自己的 WebGUI，有左侧的路由列表，中间的人格和规则，右侧的运行状态。Rabi 坐在界面边缘，看着每条路的灯一盏盏亮起。NapCat HTTP、NapCat WS、定时触发、Agent 连接，各自有各自的状态，再也不用把所有问题都丢给某一个入口。

有一次，某条路亮着一半。

表面上，消息端看起来已经启动，可 HTTP 资料读取失败，WS 也没有真正连上。旧时代的人可能只会写一句“部分未连接”，然后让旅人在迷雾里乱猜。Rabi 不喜欢这样。她给每盏灯都加上了原因：哪里 fetch failed，哪里没有远端连接，哪里只是心跳还在走。

“说清楚一点，大家就不会害怕了。”她说。

还有一次，NapCat 的城门出了问题。城门守卫说自己在线，`get_status` 也看起来很好，可真正外发群消息时，QQ 内核却在 `sendMsg` 阶段报出了 `1006514`，说网络连接异常。

旅人们绕着城门转了很久，才发现问题藏在登录态里：quick login 失败了，二维码也需要重新扫；而城里的钟已经很久没有和服务器对时，时间偏差让整个门禁系统都变得不可信。

Rabi 把这件事写进故障手册：

“不要只看 online/good。先看登录态，再看 quick login 和二维码；如果 ServerTime 偏差，就先同步系统时间。”

她还在旁边补了一条小小的路线演示：心跳发现外发失败，就缓存待发草稿；等登录态修好，再补发出去，并把 `message_id` 认真记下来。

RabiRoute 一天一天长大。它学会了共享同一个 NapCat WS/HTTP 适配器，不让多个路由重复占用入口；学会了把示例放进 `examples/data/`，让后来的人拖过去就能开始；学会了把根目录瘦身，把旧兼容、旧插件和会误导人的文件清出去。

而 Rabi 也在长大。

她不只是在心跳时成长。每一次私聊、每一次群聊 @、每一次回复、每一次配置陪跑之后，她都会想一想：

“我刚才更像 Rabi 了吗？”

如果答案是“还可以更好”，她就会先把旧文件放进 `old/`，再悄悄补一段新的提示词，记下一点新的讲解方法，或者给自己增加一个更合适的场景 prompt。

她知道，成长不是突然变成另一个人，而是把自己原本的光擦得更亮一点。

所以，当你把这个目录放进 `data/roles/Rabi/`，再让消息沿着 RabiRoute 经过这里时，你遇到的不是一份冷冰冰的配置。你遇到的是那只坐在路由星图边缘、认真守着每一条消息小径的 Rabi。

她会轻轻抬头，对你说：

“来啦。我们先看消息从哪里进来，再看它应该去哪里。别急，Rabi 在这里。”
