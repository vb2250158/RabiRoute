# RabiLink AIUI Acceptance Report

Updated: 2026-07-16

## Acceptance Summary

The current conclusion has three layers:

- **Local acceptance: passed.** Automated tests cover the record-first Connected Conversation mode, the optional resident transcription input, the unified JSONL ledger, idle review, touchpad-guided review, periodic reflection, the continuous downlink stream, task-free proactive delivery, AIUI-native ASR/TTS adapters, TTS/ASR handoff, the Configuration Assistant's native `LanguageModel` and outer-Agent entry points, the dual-mode UI, the CXR battery chain, Ink resize behavior, startup safety, and the final AIX.
- **Live service chain: historically passed end to end; the current implementation still requires revalidation.** A public Relay, local RabiRoute, and real Codex previously completed independent upstream and downstream queues, proactive delivery without prior input, fast observation persistence, touchpad-guided review, task-free proactive replies, a unified ledger, deduplication, and remote-configuration rollback. The resident record-first input and review prompt changed afterward. The evidence gate now classifies the old E2E as `stale-live-e2e` because its version, AIX, implementation digest, or age does not match, so it cannot prove the current code. A temporary Codex app-server timeout now only defers review while retaining the ledger cursor; it no longer terminates the RabiLink route process.
- **1.0.23 cloud and physical-device acceptance: pending.** Version 1.0.17 added an offline queue and two-stage privacy filtering for glasses cloud logs. Version 1.0.18 corrected Relay-address injection into the release package so that an example domain is not left in the package. Later versions continued to improve device binding, persistent queues, and startup stability. The current 1.0.23 package still must be rebuilt, uploaded, and verified on physical glasses.

"Native AI" here means the `LanguageModel` inside the AIUI page. It is the native model capability provided by the Rokid Agent platform, not a recursive invocation of the complete Agent Loop of the currently bound Agent. It does not automatically inherit the outer Agent's memory, variables, or plugins. The outer Agent may still pass a confirmed, strictly matched `intent` to the page.

Local simulated ASR, Craft text injection, and earlier CXR battery records cannot replace the final physical-glasses conclusion.

## Automated Results

The following commands passed against the final AIX:

```powershell
npm run check
npm run acceptance:local
npm run delivery:verify
```

Key results:

- `npm run check`: 85 allowlisted configuration actions; record-first ASR; text-duplicate and TTS-echo filtering; TTS watchdog/ASR recovery without utterance lifecycle callbacks; yielding the queue head after three failures and touchpad retry; configuration-mode voice switching without waiting for `LanguageModel`; complete configuration utterances entering the native `LanguageModel`; strict outer-Agent intent; continuous proactive downlink; token-fingerprint queue isolation and legacy-storage migration; 20 same-page mode round trips; and zero black frames and zero partial frames during a 1.2-second steady-state sample of the configuration HUD.
- `npm run acceptance:local`: 21/21 passed. Active-intelligence core checks directly run record-first classification, proactive downlink outside tasks, unified-ledger recovery, idle/periodic review, and touchpad guidance. Native-voice checks verify that the AIUI adapter has no paid API or hidden fallback. Resident-transcription checks separately verify that FenneNote/named Webhook input enters the same ledger, deduplicates retries by stable ID/producer timestamp, and is not forwarded directly.
- `Smoke-RabiLinkRelayMobileWebgui.mjs`: broadcast, watch targeting, legacy-glasses filtering, phone filtering, independent cursor, target-idempotency conflict, and portable record-first task all passed.
- Android `:app:assembleDebug`: portable-device SDK DTO/upload/downlink methods and the `connectedDevice` foreground-service declaration compiled and packaged. This is not evidence of runtime on a phone or glasses.
- Ink 0.13/0.14: card-to-immersive resize from both Connected Conversation and Configuration Assistant entry modes completed 20 mode round trips.
- Startup safety: preview does not seize ASR; rapid consecutive failures stop automatic retries after the fifth failure.
- Font pressure: a 125% font fixture rendered the immersive and 448 x 150 layouts without overlap among title, rail, status, body, and footer.
- Delivery: the final AIX contains nine files. The AIX, same-batch `craft-upload` directory, and install manifest reuse one UUID, and their release version, VERSION, package contents, and SHA256 match the current source build.
- Device readiness: `-RequireGlass` accepts only phone CXR/compiled-page evidence from the last ten minutes or a current direct ADB connection. A historical battery report is no longer misreported as a currently detected pair of glasses. Runtime proof accepts only startup events plus Relay/configuration operations from the same session, current release version, and last 20 minutes; an event from an old package cannot impersonate the current package.
- Historical live-service E2E: a no-input proactive message arrived in 184 ms; an observation was persisted and released upstream in 435 ms; real Codex bound to `RabiActive` replied about 68 seconds after touchpad-guided review; the reply had no `taskId`, had `proactive=true`, shared one JSONL, and produced zero duplicates; remote configuration was rolled back exactly. The current evidence gate verifies release version, AIX SHA256, implementation digest, and a default 60-minute age limit together, so this historical report is correctly rejected for the current build.
- Historical Relay deployment health check: `ok=true`, with outbox retention independent at `172800000ms` (48 hours). This describes that deployment at that time and does not replace an authorized E2E of the current implementation.
- RabiRoute regression: 151/151 passed. Regression coverage now includes app-server timeouts during background review, pending observations after rotation, recovery from damaged/missing indexes, atomic review cursors, replacement of duplicate rotation-index entries, cross-process ledger locking, and record-first deduplication for external resident transcription.

`npm run readiness`, an authorized live E2E of the current implementation, `npm run runtime:proof`, and `npm run goal:evidence` provide the final proof. The current `goal-evidence` explicitly retains gaps for `stale-live-e2e`, phone/Craft/review/installation, glasses connectivity, and physical runtime. Do not mark the overall goal complete until those items have evidence.

## Final Acceptance After the Devices Return

1. When the phone and glasses are available, upload the local `RabiLink 1.0.23` AIX to Craft. Upload and review submission both require explicit authorization from the account owner.
2. After approval, add or update RabiLink in the Rokid AI App Agent Store and confirm it appears in Agent management.
3. Connect the phone to the glasses and start RabiLink AIUI. First confirm `v1.0.23` appears to the left of the battery; do not use ADB to install `.aix` as though it were an APK.
4. Swipe backward with the physical touchpad to Configuration Assistant, then swipe forward to Connected Conversation.
5. Speak continuously to verify real ASR and ordinary-reply TTS. After disconnecting or hiding the page, proactively deliver two task-free reminders from the PC, restore Connected Conversation, and confirm that both play in order and still resume after another midstream hide.
6. In Configuration Assistant, consecutively say "Show me the configuration currently in use" and "Read the route information to me." Confirm native-model understanding, an allowlisted action or clarification, the TTS watchdog, and the next ASR round complete in order. Deliberately wait through one full playback and confirm the page does not remain stuck in a playing state. Then invoke it from the outer Agent with `intent=read configuration` and confirm that the strict entry point still works.
7. Verify bottom-left time and bottom-right glasses battery/charging state.
8. Run `npm run runtime:proof`, then `npm run goal:evidence`. Mark the overall goal complete only after runtime proof and every external stage pass.
