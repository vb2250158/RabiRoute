# 版本更新

## 0.1.20 - 2026-07-21

### 计划会话任务完成 Hook（实验）

- Codex 处理端配置新增“Hook 管理”，显示会话入口、推理期刷新和计划任务会话完成通知三组开关，并明确列出 `SessionStart` / `UserPromptSubmit`、`PreToolUse` / `PostToolUse`、`Stop` 的触发时机。三组默认开启，开关只控制 Manager 响应，不修改插件注册。
- 计划新增可选 `taskBinding`，按完整 Codex `sessionId + workspace` 绑定执行任务，并可用 `completionHook.gatewayId` 指定同人格的提醒 Route。
- 计划存在 `taskBinding` 但省略 `completionHook` 时，完成通知默认开启；可用 `completionHook.enabled=false` 单独关闭，目标 Route 也可用 `codexHooks.planTaskCompletionEnabled=false` 拒绝通知。
- Rabi Codex Context 插件升级到 0.4，新增官方 `Stop` Hook 转发。Manager 使用 `turn_id` 与 `last_assistant_message`，按 `sessionId + turnId` 持久化去重，并通过现有角色面板、Forwarding、AgentPacket 和 Agent adapter 链投递提醒；成功时 Hook 无输出，失败只返回非阻塞系统警告。
- Hook 不读取 transcript 猜结果，不自动推进计划、修改步骤或写记忆。workspace、人格、gateway、多计划绑定和源目标同会话冲突均失败关闭。
- AIUI 高级 Gateway 配置同步覆盖 `codexHooks` 三组开关，但不会代理计划审批意见；审批会写审计记录并通知真实 Agent，仍需用户在 RibiWebGUI 或托盘查看完整计划后显式提交。
- 已覆盖计划 Schema、Manager HTTP、Stop 去重/失败状态、插件 stdout/警告以及 RolePanel timeline/触发交接链测试。尚未在两个真实 Codex Desktop 任务之间完成可见性验收，因此成熟度保持实验；插件更新后需在 `/hooks` 重新审阅并信任变更。

### 计划审批意见与双端一致界面

- Rabi Manager 现在统一派生计划显示状态、审批能力和“状态优先、同状态按更新时间”的顺序；RibiWebGUI 与 Qt 托盘使用相同的标题/状态、当前步骤、关键词、完整步骤和进度层级。
- 新增 `GET/POST /api/roles/:roleId/plans/:planId/feedback`。WebGUI 或托盘可在需要审批的计划卡片中填写建议，Manager 将意见按 `planId/stepId` 写入独立 JSONL 审计记录并经角色面板链通知 Agent；记录成功但投递失败时保留草稿并用同一 `feedbackId` 重试。
- Agent 从 QQ 等入口获得审批后可调用同一接口以 `record_only` 补记用户意见或处理结果。审批记录不会直接推进计划、通过外部动作或修改状态；只有 Agent 后续显式 PATCH 计划才会改变步骤。

### 主动智能用户状态与情景识别设计

- 主动智能设计新增版本化 `CurrentUserState`，把时间、环境、活动、情绪、身体、注意力、社交、任务、交互、设备和安全拆成独立维度。
- 推断值携带置信度、证据、观测时间、过期时间和用户确认；确定事实不伪造模型分数，情绪使用愉悦度、激活度和压力等多轴值。
- 情景识别成为设备事件与意图理解之间的正式层，明确区分观察事件、用户个体模型、当前状态、情景和意图。
- 情景支持宏观/活动/微观层级、一个主情景、多个并行情景、候选解释、生命周期、证据冲突和用户纠正。
- 介入不再硬编码“低打扰优先”或“始终高主动”。状态、情景、Agent 人格、用户模型、收益、时效、置信度和风险共同参与决策。
- 决策结果包括不打扰、后台准备、轻提示、主动建议、请求确认、直接行动或紧急介入。
- RabiLink 需求文档明确设备只提供观察，PC 上下文层维护可重建快照，RabiRoute 负责安全记录和投递，目标 Agent 人格做最终介入判断。
- 统一状态服务、情景引擎、用户模型存储、控制面和实体场景验收仍是目标合同，尚未全部实现。
- 主动智能新增四层用户个体模型：稳定性格假设、按情景学习的偏好、当前心理状态和情景心理特征。
- 五因素性格只作低权重连续维度，明确指令和用户纠正优先；Agent 人格与用户性格分开维护。
- 设计融合情感环形模型、自我决定理论、任务负荷和 DIAMONDS 情景特征，并禁止临床诊断、操纵式优化和单次行为定型。
- 用户模型需要可查看、纠正、删除、暂停学习和导出。

### 常驻 ASR 原始消息与完整声纹上下文

- 主机原始消息新增整段 RMS，并与峰值、采集时间、来源、音频格式、ASR/声纹分段一起贯穿人格语音历史和会话上下文；RMS/峰值只作为音频事实，不参与主机身份判断。
- 修复 `pre_roll_ms=0` 时触发 VAD 的首块有声 PCM 未进入语段的问题，避免句首丢失以及只剩尾部静音时整段 RMS 错误为零。
- `scripts/test_multivoice_voiceprint.py` 现在可经回环 RabiSpeech API 自己生成多个匿名 TTS 源，并可选把组合 WAV 交给真实会议 ASR 检查自动 diarization。测试发现 `paraformer-v2` 为三个完整时间 turn 只分配了两个 speaker 标签；声纹层因此改为保留原始 `speaker`、按不连续 turn 生成独立 `speakerLabel` 和 embedding，再让 cluster 决定是否同一声音。修复后同一组合音频仍显示 Provider speaker `2/3`，但本机 voiceprint 已正确得到 `3/3` 并标记 `providerMergeCorrectedByVoiceprint=true`。报告不保存正文、声线名或原始声纹 ID，仍不能代替真人正式校准。
- 多声线预检不再把命令行模型名直接当成有效协议值；运行前读取 `/v1/models`，接受完整 ID 或唯一短名，跨 Provider 重名和不可用模型明确失败。当前三声线真实闭环再次得到 Provider speaker `2/3`、本机 voiceprint `3/3`、19 秒音频覆盖率 `100%`，仍保持 `formalValidationEligible=false`。
- Android PCM 断网恢复改为事件优先：系统默认网络 `onAvailable` 与既有 RabiLink SSE 恢复会立即唤醒待确认音频，服务端临时失败才使用一次性 1–30 秒退避。每个待确认块新增稳定 `chunkId`；RabiSpeech 按稳定设备只保留“最后一个已接收 chunk”的 ID 与 PCM SHA-256，ACK 丢失后即使 Android 长时间断网并换了临时 `sourceStreamId`，同一 PCM 也不会再次进入 VAD/ASR。Android 只保留一个待确认块和有界最新 PCM，长时间断网会舍弃过旧声音并在恢复后追上实时流，不无限占用内存、也不把原始录音偷偷落盘；进程死亡时未确认 PCM 明确丢弃。文字、控制和媒体继续使用磁盘可靠队列。
- Android 的事件优先恢复进一步收紧：新增可测试的网络事件门。系统明确无网络时，SSE 连接和可靠队列发送会休眠到 Connectivity callback，不再每 2.5 秒唤醒；仅“网络存在但 Relay/服务失败”才按 1–30 秒退避。眼镜播放则在 `PLAYBACK_BEGIN` 后等待主线程确认录音已暂停，再允许播放线程接收 PCM；无效新 BEGIN 不再把失败误记到上一条终态消息，Activity 销毁会对未完成播放发出 `playback_failed`。
- 为避免少数 Android 厂商漏发已注册默认网络回调后永久停在离线事件门，前台服务新增仅在“系统已知离线”期间运行的五分钟安全检查。它只读取 OS 当前网络，一旦恢复立即停止并唤醒既有 SSE/可靠队列；随后仍由 SSE `ready` 触发一次持久 cursor 增量补漏。该兜底不访问 Relay、不扫描消息、不推进 cursor，事件仍是主路径。
- Android 下行 SSE 现在使用 45 秒传输停滞 deadline 覆盖“系统仍显示有网、但 TCP 已半开且收不到 keepalive”的黑洞窗口；Relay 正常每 15 秒发 keepalive，超时后只重建连接并由 `ready` 做一次 cursor 补漏，不增加业务轮询。消息连接恢复意图也与持续聆听分离持久化：已启动的文字/媒体/下行连接在进程或设备重启后恢复可靠队列，用户明确停止才关闭自动恢复。
- 新增 `npm run check:active-intelligence:physical` 统一实体环境验收状态。工具一次性读取真人声纹、双 PC、Android soak、Rokid 真机与人工观察证据，不启动测试、不轮询设备；四个领域分别输出 `missing/partial/passed/stale/invalid`，默认任一未通过即退出 `2`。合成 TTS、缺字段、过期或哈希不匹配证据不能冒充正式完成；脱敏汇总只保存证据哈希、时间与检查项，不保存 token、正文、人格、主机、设备序列号或私有路径。
- 双 PC 验收报告升级为 schema v2，功能终态 `syncPassed` 与正式实体资格 `formalAcceptanceEligible` 分离；只有显式确认两台不同实体主机的成功同步才成为候选正式证据。原始报告不再保存主机名、Manager URL、peer ID/GUID/name、人格 ID 或冲突路径，只保留数量、同步范围、传输与冲突类型计数。新增 `npm run record:active-intelligence:physical`，用 allowlist 内的逐项 `--confirm/--revoke` 维护人工物理事实；旧文件先归档，环境只保存随机证据的 SHA-256，不提供批量全真或自由文本入口。
- RabiLink 下行补齐设备事实回执和超长断网保护。Android 将 `delivered/played/playback_failed` 先写私有回执队列，再由网络/SSE 事件唤醒补传；SSE 之后仍按持久 cursor 查询一次，覆盖断网期间漏事件。手机和眼镜都只在各自 `AudioTrack` marker 到达后报告 `played`，眼镜 BEGIN/PCM/END 共用有序 Classic BT；旧无帧 PCM 不产生成功回执。Relay 持久化逐设备回执并发布 `outbox_receipt`，明确 `targetDeviceIds` 的消息在全部目标 `delivered` 前不按 48 小时 TTL 删除。文字/control/媒体可靠队列不再静默清理未确认旧项目，满载时改为显式拒绝新项目；回执终态同时作为崩溃后的本地去重证据，避免实际播完后进程被杀造成重播。
- RabiLink 下行 cursor 改为带 Relay 共享代际的不透明值。正常重启与同数据目录的多 Relay 进程共用代际；`runtime-state.json` 回滚或游标损坏时返回 `cursorReset=true`，从当前保留消息重放。Android 与 AIUI 优先按稳定 `deliveryId` 去重后保存新 cursor。新增回归先证明普通重启不重放，再故意回滚序号并复用同一个 `out-...` 数字 ID，证明新消息仍能被取回，不会因手机游标永久领先服务端而永远滞后。

- RabiSpeech 常驻 ASR 不再只向 Manager 提交纯文本；现在同时提交稳定 `recordId`、采集/接收时间、Provider、模型、语言、时长、采样率、通道、来源设备、完整说话人分段，以及 Provider 可用时的逐词时间、概率/置信度和说话人标签。RabiRoute 先写主机级 `data/speech/messages/YYYY-MM-DD.jsonl`，再按逻辑消息端分发；逐词证据继续进入人格 `voice-transcripts.jsonl` 和统一会话账本。
- 每个 Route 继续按 NapCat 相同的“消息端 → Route → 人格”关系写入绑定人格：本机麦克风和普通 Rabi 语音客户端只命中 `speech`，手机音频流只命中 `rabilink`；不同人格各有 `voice-transcripts.jsonl` 与 `conversation/current.jsonl`，两份文件都保留来源设备、临时流 ID、音频格式、Provider/模型和完整说话人证据，同一人格被多 Route 命中时不重复记录。手机回复默认按 `sourceDeviceId` 回原设备；无对应订阅时只保留主机通用消息。
- 主机语音原始记录到 Route 事件的完整字段映射收拢到 `src/routing/speechIngressForwarding.ts`，正式 CLI 与闭环回归共用同一入口。修复人格首次收到语音时，刚写入的 `voice-transcripts.jsonl` 被初始化器当旧历史导入、导致 `conversation/current.jsonl` 同一事件出现两次的问题；现在先写统一会话账本，再写兼容原始历史。闭环回归会继续把同一个 AgentPacket `replyContext` 交给真实 Outbox/Relay 发布器，确认回复只使用稳定 `sourceDeviceId`、不会误投给临时 PCM `sourceStreamId`，并把出站回复追加回同一人格会话。
- 未知声纹簇新增服务重启持久化回归：`speaker-embeddings.json` 中的 cluster 在 RabiSpeech 重启后继续复用；原有全天裁剪回归同时保证低频说话人的簇原型不会被高频声音全部挤出。
- 新增真实 `src/index.ts --speech-message` 子进程回归，并修复其发现的来源分类错误：`routeProfileId` 原本被旧类型判断当成“角色面板”证据，导致带选定人格的手机语音 AgentPacket 错写 `targetType=role_panel/adapterType=rolePanel`。现在 `routeProfileId` 只选择 Route，手机语音保持 `rabilink` 来源并可沿普通回复 API 回原设备。对称的 PC 麦克风子进程回归同时锁定 `targetType=voice_transcript/adapterType=speech`，并确认不会携带手机下行设备或临时流目标。
- 人格会话记录保留不透明声纹 ID、未知簇、分段标签、分数与声纹判定证据。进入主机通用消息和人格会话前会统一删除主机人物名称、资料/候选资料 ID 与已验证人物标志，读取旧会话时也会再次清洗；主机不判断声纹是谁或谁是用户，对应人格结合自身关系和上下文解释身份。
- 主机通用消息、人格账本与 AgentPacket 新增 `sourceHostId/sourceHostName`。人格可通过 `GET/PUT /api/roles/:roleId/voice-identities` 独立维护追加式 `voice/voice-identities.jsonl`；身份按“处理主机 + 声纹 ID”分域，`isUser` 没有系统默认值，修正和删除使用新事件/tombstone，随多电脑人格同步自然合并。关系事件现在记录 `supersedes` 父事件头；多 PC 并发判断不再按行序静默覆盖，GET/AgentPacket 暴露冲突字段和候选分支，后续人格 PUT 会显式收敛全部当前头。
- 新增 `GET /api/roles/:roleId/voice-transcripts` 人格级只读视图：按时间、归档和 `user/other/unknown/conflict` 筛选一天内的语音，把会话账本与当前人格声纹关系在读取时联结并逐分段分类；派生称呼和 `isUser` 不回写原始消息，关系修正后查询结果立即更新。查询现增加用户/他人/未知/冲突的分段数与说话人时长、覆盖率和未解决声纹汇总，`matchedCount` 与 summary 不受明细 `limit` 截断。
- WebGUI 人格页新增“人格声纹归类”：展示最近 24 小时覆盖率、用户/他人/未知/冲突统计、未解决声纹缩写和已归类关系，并提供“这是我 / 其他人 / 清除判断”。页面使用 `includeDetails=false`，不接收或展示转写正文；本机 `records_changed`、人格关系写入和多 PC 文件事件触发单次刷新，事件流重连时补查一次，不做固定轮询。新增“标记下一段”辅助流程：用户主动开始后，下一次语音记录事件只高亮本次新出现且可跨电脑定位的未归类声纹，仍需人工确认，单一候选也不会自动认定为用户。同步修复 Manager 总路由未放行 `PUT /api/roles/:roleId/voice-identities` 的问题，人工和 Agent 修正现在能到达真实追加事件入口。
- 声纹正式校准脚本现在和模型探针一样从自身目录加载 RabiSpeech、`.deps` 与本机运行库，可直接从仓库根目录或任意 cwd 执行；新增子进程回归测试，避免 Agent 按文档运行时因缺少 `PYTHONPATH` 失败。本机探针继续验证 ERes2NetV2 可输出 192 维 embedding，但没有私有同人/异人报告时仍保持 `validated=false`。
- 新增私有声纹校准语料采集工具：可列出输入设备、直接录制或导入 WAV、统一 16 kHz mono、拒绝低 RMS/严重削波样本、原子维护并归档 `speaker-cases.json`，以及在跑模型前按 policy 显示数据规模缺口。默认目录处于 Git 忽略区；工具不接触人格关系，也不会把采集标签解释成用户或自动开启 `validated`。
- 使用显式启用的 `dashscope-qwen/qwen3-tts-instruct-flash` 完成一次 32 条、6 个匿名系统声线的合成声纹预检。当前 ERes2NetV2 在 threshold `0.72` / margin `0.06` 下通过示例 policy，得到 EER `2.804%`、FAR `0.236%`、FRR `5.556%`、已知识别率与未知保留率均为 `100%`。运行元数据明确标记 `dataset_kind=synthetic_tts`、`formal_validation_eligible=false`；合成结果不写入 `validation_report_path`，不改变 `validated=false`，也不替代真人、真麦克风和真实环境校准。
- 正式声纹门禁改为真人数据集失败关闭：采集 manifest 默认无正式资格，只有显式 `init --confirm-real-person-recordings` 才标记 `real_person_private`；benchmark 报告新增 dataset manifest/policy SHA-256 与总门禁结果，合成 TTS、旧报告或未声明数据集即使指标通过也以退出码 `2` 拒绝。运行时同步核对数据集类型、资格、完整门禁、dataset/policy/model 哈希和阈值，不能再靠手工设置 `validated=true` 绕过。
- 新增 `npm run check:rabispeech:tts-loop` 运行时闭环验收。工具先订阅 RabiSpeech/Manager SSE，再按动态模型发现执行 TTS → 16 kHz mono WAV → ASR → 声纹证据 → 双层记录查询；只有 TTS/ASR 两个 `records_changed` 终态到达后才各查询一次记录，不做轮询。默认优先本地模型，API Provider 必须显式授权；工具不启停麦克风、不播放、不投 Route，脱敏证据不保存测试正文、声线或原始声纹 ID，并固定标记为非正式校准。
- 修复 Manager 语音 SSE 客户端断开时，上游 fetch 的 AbortError 可能从 `Readable.fromWeb().pipe()` 变成未处理流错误并终止进程的问题。新的 `speechEventProxy` 独立管理每条客户端/上游生命周期，正常消费断开错误，并在上游 Content-Type 不是 SSE 时失败关闭；旧 8790 返回 WebGUI HTML 不再被验收工具误当成事件流。

- AgentPacket 现在只在当前路由消息明确要求多电脑人格同步时，向绑定人格注入同应用 peer 查询、当前人格一次性同步和冲突终态合同；默认不扩大到全部人格，peer 不唯一时要求确认，普通消息不增加提示或轮询。这样“由 Agent 调接口同步”成为当前任务可发现能力，而不依赖 Agent 偶然读到专题文档。
- 新增 `routing/agentCapabilityHints.ts` 作为按意图能力提示归口。用户询问声纹、说话人、全天录音中哪些是用户/他人发言时，AgentPacket 只为当前任务注入人格 `voice-transcripts` 时间范围查询、`user/other/unknown/conflict` 汇总和 `voice-identities` 追加修正合同；不确定保持 unknown，主机候选/分数不能直接变成人格身份，普通消息不增加提示或覆盖率轮询。
- 语音投递新增持久终态 receipt：同一 `recordId + Route` 的并发提交、HTTP 重试和补交只投递一次，成功/仅记录结果会直接复用；失败不写成功 receipt，修复 Desktop owner 或 IPC 后仍可重试。
- 主机原始语音的 `recordId` 去重与日文件追加现在共用跨进程锁，Route receipt 日文件也串行写入，不再依赖单进程时序。新增 `GET /api/speech/messages`：可列出最近通用语音消息，或按 `recordId` 查询完整原始记录与逐 Route 终态。
- Android 手机/眼镜现在与独立 Rabi 语音客户端一致，只持续传输有序 16 kHz mono PCM；手机端不再暴露或执行 VAD、静音切句、ASR 与声纹。Relay 新增受限 `audio-streams/rabilink/start|chunk|stop`，目标 PC RabiSpeech 完成 VAD/切句/ASR/声纹后自动写主机通用语音入口，并按冻结的 `routeProfileId` 投给 RabiLink/手机消息端。chunk 序号必须连续；Android 在 PC 确认前保留 pending PCM，同序号同内容可幂等重试且不会重复进入 ASR。PCM 上传改为有界执行队列，弱网长期阻塞时丢弃过期排队音频并生成新的临时 `sourceStreamId` 重连，避免手机/眼镜录音回调无限占用内存。`start` 和每个成功 chunk 会重置一次性 15 秒到期事件；无数据到期后自动回收旧虚拟输入并恢复之前的音频源，不再每 2 秒扫描流状态。`/api/rabilink/speech/messages` 只保留兼容/调试用途，普通回复默认回原 `sourceDeviceId`。
- 消息端类型改由采集入口强制拥有：`audio-streams/rabilink/*` 永远生成 `rabilink`，局域网远程声卡永远生成 `speech`；客户端请求体或 WebSocket hello 不能把两种来源互相伪装。

- Android 正式会话后端移除旧的整段音频磁盘队列、直接 `audio/transcriptions` 请求和手工发布语音消息旁路；ASR 模型、语言、VAD 阈值和静音窗口也不再属于移动端设置。手机/眼镜现在只有有序 PCM stream 一条输入真源，ASR 缓存与元数据只在 PC RabiSpeech 产生；Android 仅保留下行 TTS 的 24 小时播放缓存。移动端架构审计新增反向门禁，防止这些旧入口再次回流。
- 修正主 App 眼镜输入把共享字面量 `rabi-glass` 当作下行设备的问题：眼镜 PCM 与媒体现在使用实际轮询 Relay 的伴侣后端 `deviceId` 作为 `sourceDeviceId`，以 `sourceDeviceKind=glasses` / `channelType` 保留物理来源；Agent 回复因此会先准确回到原伴侣手机，再由手机送往眼镜。

### 多电脑人格数据同步

- 新增 `src/personaSyncManifestIndex.ts` 作为 manifest 派生状态唯一拥有者。索引不保存文件正文或绝对路径，也不参与多电脑同步；重启复用、文件事件单路径重算、事件不可用的一次性查询校准均有自动化回归。事件驱动门禁同时禁止把全目录 `walkFiles` 哈希重新塞回 manifest 查询。
- 同一 RabiLink 应用 token 下的 PC worker 现在登记 `persona-sync` 能力和专用局域网数据面 listener URL；Relay 新增同应用 peer 查询。同步器优先局域网直连，失败后通过受限 Relay proxy 复用现有 PC worker 转到目标回环 Manager。专用 listener 默认使用系统分配端口且只开放 manifest/file/merge，不再要求把完整 Manager/WebGUI 绑定到局域网。
- Manager 新增人格 manifest、文件读取、单文件 merge 和整机同步 API。JSONL 按稳定记录身份并集合并；普通文件根据 per-peer 共同哈希快进；双方同时修改时不覆盖，远端内容写入 `data/persona-sync/conflicts/`，旧版本写入 archive。同步终态新增 `fileConflicts/semanticConflicts`：人格声纹 JSONL 即使文件合并成功，未收敛分支也会在同一次响应中计入冲突并返回 `409`，无需 Agent 轮询覆盖率。
- 同步明确排除锁、临时文件、符号链接、可再生 TTS 缓存和超过 16 MiB 的单文件。普通文件建立共同基线后，任一侧缺失会作为删除经 LAN 或 Relay 双向传播，删除前归档旧文件；删除与另一侧编辑并发时不会静默扩散，而会生成带 `remoteDeleted`、peer 与基线哈希的冲突证据。首次同步缺失仍按新增处理；WebGUI 与事件驱动自动对账不会改变这些保守合并规则。
- 同一 peer/人格的并发同步改为 single-flight；peer 基线状态与文件替换使用锁和原子写。读取、创建与 merge 会检查完整父路径链并拒绝符号链接/Windows junction，避免通过人格目录链接读取或写入外部文件。
- 普通文件共同基线现在按“应用 token 哈希作用域 + 对方稳定设备 GUID”分域，不会跨 RabiLink 应用误用旧基线。`conversation/` 合并复用消息上下文锁，语音记录和人格声纹关系复用自己的文件锁，避免同步替换与在线 Agent 追加交错。
- 新增仅本机可用的人格文件冲突列表、远端证据读取和解决 API。Agent 可选择 `keep_local`、`use_remote` 或提交 `use_merged` 内容；Manager 以 `expectedLocalHash` 拒绝过期覆盖，替换前归档本地版本，把原冲突移入 `resolved-conflicts/` 并写解决审计。解决完成后会按冲突里的 peer/远端哈希即时经 LAN 优先、Relay fallback 发布结果；只有两端仍匹配证据才返回 `publish.status=published`，离线或变化时返回 `not_published`，不以后台轮询伪装收敛。专用 LAN listener 与 Relay allowlist 均不暴露冲突控制。Windows 锁文件竞争返回 `EPERM` 时现在会按已有锁等待，避免人格会话在线写入期间同步错误失败。
- 默认 `npm test` 现包含 Relay 人格同步隔离回归：它会在同一管理账号下创建两个不同应用，明确验证 peer 查询只返回同应用 PC，且双方都无法通过人格同步 proxy 中转到另一应用的 PC；同 token 的发现、LAN URL 清洗和 Relay proxy 成功链继续同时覆盖。
- 默认门禁同时运行 Relay 事件 hub 与手机语音代理闭环：事件只唤醒同应用、目标设备订阅者，不会跨应用广播；手机 `start → binary chunk → stop` 会把稳定 `source_device_id` 与临时 `stream_id` 原样交给目标 PC RabiSpeech，避免测试遗漏后再次把回复目标绑到音频流。

### 事件驱动主链与空转性能

- 事件架构门禁在递归入口跳过 `.pytest_cache`，避免不可读测试缓存让源码扫描提前失败；轮询禁令和受控例外清单不变。

- Relay 新增 `/api/rabilink/events` SSE。PC worker、Manager WebGUI/语音代理与 Android 下行现在收到 `task/webgui/speech/outbox_available` 后才即时领取；cursor 只在重连时补漏。没有 SSE 的 AIUI 和旧客户端仍可传有界 `waitMs`，但服务端只等待同一内部事件并在订阅后复查一次，不扫描业务队列。活跃 PC SSE 连接同时成为同应用发现的在线真源；重连交叠时只有最后一条连接关闭才立即离线，不再靠周期心跳续状态。
- RabiSpeech 新增 `/v1/events`，Manager 通过 `/api/speech/events` 转发麦克风电平/状态、播放队列、音频流与 `records_changed`；记录事件只在 ASR/TTS 成功落盘后发出。WebGUI 删除 400ms/1s/3s/15s 语音状态轮询与 5 秒记录轮询，SSE 重连只做一次快照补漏。

- 事件门禁现在扫描后端、WebGUI、Android 主 App、RabiSpeech、桌面语音客户端、托盘和 AIUI 生产源码；固定 interval/fixed-delay 业务调度默认失败，只有带明确原因的 SSE keepalive 可通过。Android 录音停滞改为基于最后成功读取的一次性 45 秒 deadline；Relay 附件和 RabiSpeech TTS 缓存都改为“启动扫描一次 + 最早到期 deadline”，不再周期扫描目录。门禁会列出仍保留的 DashScope 远端异步任务、AIUI 前台长等待/电量刷新和小米健康 ADB 低频采集例外。
- Remote Agent 认证超时现在按 WebSocket `CONNECTING/OPEN` 状态安全终止连接，并吸收预期的握手中止错误；超时只永久结算为明确失败，不再因关闭竞态抛出未处理异常或随后保存迟到认证。

### ASR 热投递、人格关键词与真实送达状态

- Route 新增 `speechPushMode=hot|keyword`，WebGUI 用一个“热投递”开关编辑：开启时每段 ASR 完成立即投递；关闭时所有转写仍记录，只在命中当前人格 `speechTriggerKeywords` 时唤醒 Agent。人格关键词为空时保持“只记录”，不暗中回退热投递。
- `speechTriggerKeywords` 归人格 `personaConfig.json`，可以配置人格名、常用称呼和唤醒词；同一人格被多条 Route 复用时共用该列表。
- `POST /api/speech/messages` 从旧 `202 accepted` 改为同步终态：常驻麦克风省略 `routeId` 时广播并返回每个订阅 Route 的 `deliveries[]`；`200 delivered` 表示至少一个 Desktop 已收到，`200 recorded` 表示只记录或当前无订阅，所有订阅失败才返回 5xx。显式 `routeId` 只保留给调试/兼容调用。语音监视面板不再用其他 Route 的成功冒充当前 Route 送达。
- 已匹配的普通消息端事件仍直接进入 Desktop owner：活跃 turn 用 `steer`，空闲用 `start`。Heartbeat 的 `heartbeatSkipWhenAgentBusy` 保留为独立例外，不影响普通消息。

### 统一双向消息账本与分端上下文

- RabiRoute 保持稳定 adapter id `codex`：OpenAI 是 provider，Codex 是 agent/runtime，Desktop IPC 是 transport，Codex/ChatGPT Desktop 是任务 owner，具体 GPT 版本是目标任务自己的 model。
- 本机正式投递只通过 Desktop IPC；Desktop 未启动、目标任务无法加载或 IPC 合同失效时失败关闭，不改写用户环境，也不启动备用执行 Runtime。
- 固定线程以完整 `threadId` 为身份，并继续校验工作目录；线程名和最后会话时间只用于用户选择，避免误投递到同名历史线程。

### 模型与安全默认值

- 本机 Route 不覆盖模型、工具、沙箱或审批；这些能力全部沿用目标 Desktop 任务。兼容字段 `agentModel` / `sandbox` 不得制造另一套任务设置。
- Desktop 任务审批与工具授权仍由 Desktop owner 处理；RabiRoute 不通过第二 Runtime 模拟、放宽或替代这些能力。
- Codex runtime approval 与 RabiRoute Outbox / Action Gate 保持独立，Agent 执行获批不代表 QQ、文档、设备或外部 API 获得外发权限。

### 运行时与状态可靠性

- 本机 Route 只扫描 Desktop 安装与 Desktop IPC 就绪状态，任务发现来自 Desktop 状态库，真实消息通过 Desktop owner start/steer。
- 项目根锁定的 `@openai/codex` 仅服务空任务元数据 bootstrap；独立 Remote Agent bridge 在远端设备上拥有并锁定自己的无人值守 stdio Runtime，两者都不得接管本机 Desktop。
- Remote Agent 的 stdio client 保留并发初始化门、请求超时、连接关闭清理和 fail-closed server request；该边界不适用于本机 Desktop Route。
- Agent 状态上报增加 gateway 进程 generation 与单调 sequence；Manager 只接受当前进程且顺序更新的状态，启动、停止和退出时清空旧状态，避免旧异步结果覆盖新状态。
- Codex 扫描提供 Desktop 任务候选；实际身份以 Desktop 状态中的完整 `codexThreadId` 和规范化工作目录共同确认并持久化。

### Remote Agent bridge

- 独立 bridge 升级为协议 v3 双向 HMAC-SHA256 challenge，移除公知默认密码；未配置时每次启动生成高熵临时密码，Manager 严格验证协议版本、challenge、server proof 和 device id。
- 远端工作目录与回传文件都先经过 `realpath` 并按允许根/当前任务 cwd 校验，阻止 junction 或 symlink 越界；文件默认限制同步为单文件 10 MiB、单任务 25 MiB，Codex 默认 `workspaceWrite` 且禁止网络。
- 同一线程目标的任务串行到 terminal；恢复到活跃 turn 时有限等待，不能安全复用就创建独立线程。`turn/completed` 直接提取最终 `agentMessage` 回传，失败、中断、未知状态、system error 与 app-server exit 均失败关闭，不再依赖网络 callback 才能结束任务。
- callback 与 app-server terminal 事件去重；路径附件只允许当前任务 cwd。可选 `REMOTE_AGENT_PUBLIC_CONTROL_URL` 只接受无凭据、无 query/fragment、固定 control path 的 `ws://` / `wss://` 地址。
