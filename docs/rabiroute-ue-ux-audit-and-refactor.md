# RabiRoute 全界面 UE/UX 问题清单与重构方案

## 目标

把 RabiRoute 的所有可见界面统一成一套“Rabi 星海消息分诊台”体验：用户打开后能立刻知道后台是否在、托盘是否在、当前路由是谁、消息端和 Agent 哪里断了、下一步应该做什么。

视觉和交互参考采用“RabiRoute 人设 + AstrBot 管理台风格”：

- RabiRoute 人设负责品牌：Rabi 是轻巧、可靠、会分拣消息包裹的兔娘投递员。
- AstrBot 风格负责结构：左侧模块导航、可视化配置、插件/适配器管理、日志诊断、统计和 Trace 式运行观察。
- 最终效果不是复制 AstrBot，也不是做营销页，而是把 RabiRoute 做成一个有角色温度的本地管理控制台。

本轮重构保留全部现有功能，不改 manager / gateway 的核心协议，不把 Windows 托盘逻辑写进跨平台核心。WebGUI 仍是完整管理台；Qt 托盘/悬浮窗仍是桌面常驻入口；NapCat 插件页仍只是入口跳板。

## 界面范围

- RibiWebGUI：`ribiwebgui/src`
- Qt 托盘与任务面板：`desktop/tray-task-window`
- NapCat 插件入口页：`plugin-adapters/napcat-rabiroute/webui/gateways.html`
- Windows 启动入口提示：`Start-RabiRoute-*.bat`、`scripts/start-rabiroute-windows.ps1`

## 当前主要问题

### 1. 全局体验

- 首屏安全感不足：用户看到 manager 在跑，但不一定知道托盘、WebGUI、gateway、Agent 分别处于什么状态。
- 术语层级混在一起：Gateway、路由、消息端、Agent、人设、模板、Manager 出现在不同页面，但没有稳定的信息层级。
- 视觉风格还不够统一：WebGUI 偏浅色控制台，Qt 面板偏深色临时面板，NapCat 入口又是独立小页。
- Rabi 人设资产使用不稳定：图标存在，但没有形成一致的品牌节奏；需要“轻一点、稳一点”，不是花哨装饰。
- 缺少统一空状态：没有配置、没有任务、没有记忆、没有日志时，各界面的提示语气和下一步指引不一致。
- 管理台模块感不足：与 AstrBot 这类管理面板相比，RabiRoute 还缺少稳定的“模块入口、配置页、日志页、诊断页”节奏。

### 2. RibiWebGUI

- 总览页信息已经齐，但“当前应该看哪里”不够强。需要把当前路由、运行状态、消息入口、Agent 绑定放到更明显的位置。
- 侧栏只提供导航和路由选择，缺少当前运行状态摘要。
- 顶栏按钮密集：刷新、新增、保存和 Manager 状态并排，窄屏容易压缩。
- 路由配置页表单结构偏平，消息端、Agent 端、数据目录之间的关系不够明显。
- 人格配置页同时承担人格选择、persona 预览、变量、规则、模板变量，认知负担较大。
- 运行日志页有诊断数据，但健康原因、操作按钮、日志输出没有形成“先看故障，再看细节”的阅读顺序。
- 快速配置弹窗可用，但与主页面视觉层级没有完全统一。

### 3. Qt 托盘 / 悬浮窗

- 悬浮窗可切换视图，但视觉仍像内部调试面板，缺少和 RabiRoute WebGUI 的统一品牌。
- 视图按钮横向密集，当前 / 计划 / 记忆 / 状态之间的分组不明显。
- 内容区是纯文本，读任务、记忆和状态时扫描成本较高。
- 空任务提示已经存在，但缺少“这是只读面板，不会写数据”的更明显安全提示。
- 托盘菜单已中文化，但菜单顺序还可以更贴近用户任务：先状态，再打开 WebGUI / 面板，再目录，再退出。
- 无托盘环境 fallback 可用，但提示可以更清晰地区分“面板模式”和“托盘常驻模式”。

### 4. NapCat 插件入口页

- 入口页定位正确，但视觉与主 WebGUI 还不够像一个产品。
- “启动 Manager”的反馈较轻，用户可能不知道启动后该刷新还是打开 WebGUI。
- 插件页不应膨胀成第二套 WebGUI，只需要更明确地表达“这里是跳板”。

### 5. Windows 启动入口

- `1+1` 生命周期方向正确，但用户心智需要更强：从 Windows 启动时，应明确“这是托盘版，后台由它带起”。
- 当已有 manager 时，托盘只接入不拥有后台；这个规则应该在文案和日志中更清楚。
- 若没有托盘而 manager 在跑，用户会不安；启动器应该优先拉起托盘，托盘缺依赖时给明确提示。

## 重构原则

- 先统一信息架构，再统一视觉。
- WebGUI 保留全功能；Qt 面板保留轻量只读；NapCat 插件保留跳板定位。
- 所有用户可见中文保持清楚、短句、操作导向。
- 控制台密度优先：避免大段说明、营销式英雄区、过多装饰卡片。
- Rabi 资产作为识别和温度，不抢占诊断信息。
- AstrBot 式模块化只学结构，不照搬品牌：左侧模块、卡片化配置、日志诊断、状态筛选、插件/适配器入口感。
- 跨平台核心不写 Windows-only 行为；Windows 托盘和启动器只在入口层做便利。

## 实施顺序

1. 建立 WebGUI 通用页面骨架：页面头、状态摘要、空状态、危险操作区、运行状态卡。
2. 重构总览页：做成控制台首页，突出当前链路和下一步动作。
3. 重构路由配置页：按“消息入口 -> Agent 投递 -> 数据目录”组织。
4. 重构人格配置页：把人格、变量、模板规则分成更清楚的区块，规则列表更易扫描。
5. 重构运行日志页：先健康诊断，再连接详情，再日志。
6. 重构 Qt 任务面板：统一 Rabi 风格、分组视图按钮、强化只读与空状态。
7. 调整 NapCat 入口页：只做视觉统一和启动反馈，不扩展功能。
8. 验证 WebGUI 构建、TypeScript、PySide6 语法和启动入口。

## 验收标准

- `http://127.0.0.1:8790/` 刷新后四个 WebGUI 页面中文清晰、布局稳定。
- WebGUI 能保留新增、保存、刷新、启动、停止、重启、打开配置文件等现有功能。
- Qt 面板能显示当前 / 短期计划 / 长期计划 / 短期记忆 / 长期记忆 / 任务 / 状态，并保持只读。
- 托盘菜单全中文，退出语义符合 ownership：拥有 manager 时退出 RabiRoute，不拥有时只退出面板。
- NapCat 插件页仍只跳转 WebGUI，不成为第二套配置台。
- Windows 启动入口继续优先启动 manager + tray；Mac / Linux 继续通过 Node/WebGUI 使用核心能力。
