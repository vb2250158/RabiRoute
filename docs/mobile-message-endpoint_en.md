# Rabi Mobile Device Message Endpoint

The Rabi mobile device message endpoint is a new endpoint, separate from the Rokid AIUI / Lingzhu Agent MCP integration. The phone is the complete client and reliable backend. Glasses are optional microphone, speaker, HUD, camera, and touchpad peripherals. Without glasses, the phone still supports login, chat, continuous capture, ASR/TTS, attachments, notifications, configuration, and proactive messages.

## Initialized experience

- Before initialization, the app shows global RabiLink login, default Rabi PC, speech-model, and glasses authorization settings.
- After initialization, a QQ-style conversation list is the default. Each row shows an avatar, contact name, latest message, time, and per-conversation unread count. Tapping a contact opens chat; Back returns to the same list so another persona can be selected.
- Contacts come only from Routes that expose the `rabilink` message adapter. Wearable-health and other non-chat Routes are never treated as personas. A disabled RabiLink Route explains why it cannot chat and links to configuration.
- The detail header contains only Back, the current identity, and a trustworthy connection state. Messages are grouped by date, keep sender/time outside bubbles, use explicit speech/configuration/file labels, and open local attachments.
- Attachment, composer, and Send controls use one 52dp component height. Multiline input, keyboard Send, and per-conversation draft restoration are supported.
- Each conversation owns its read position, so opening A never clears B. Legacy messages without a Route are migrated once to one deterministic conversation instead of appearing under every persona.
- Text, microphone ASR messages, Agent TTS, images, video, standalone audio files, and arbitrary files share one private phone chat ledger. Attachments work in both directions and can be opened on the phone.
- Configuration no longer shares the normal chat composer. Known fields are edited in Settings or remote WebGUI; unknown fields use a separate Configuration Assistant launched from Settings. Marked requests still use the selected Rabi PC, action gate, and success/read-back requirement.

## Foreground service and notifications

After connection, `RabiConversationService` owns the downlink cursor, durable queues, and phone/glasses I/O. It exposes two ongoing notifications: one opens chat, and one sends an immediate `rabilink.review_request`, matching a single AIUI touchpad tap. Ordinary and proactive Agent deliveries use separate normal notifications.

Agent notifications are stable per conversation and carry `routeProfileId`. Tapping one uses `singleTop` to open the exact persona; Back returns to the conversation list. A newer message updates the same conversation notification, and opening the detail marks it read and clears it. An app setting controls immediate Agent TTS playback. When disabled, the WAV remains in private chat storage and can be played by tapping its bubble.

## Phone and glasses modes

- Phone mode continuously captures 16 kHz mono PCM in an Android microphone foreground service and uploads ordered, roughly 500 ms chunks through the restricted `audio-streams/rabilink/start|chunk|stop` endpoints. Android owns no VAD, utterance segmentation, ASR, or voiceprint logic. The target PC treats the stream as a virtual remote microphone; RabiSpeech performs VAD, segmentation, ASR, voiceprint processing, and automatic message submission. Results use `messageAdapterType=rabilink` and `channelType=rabilink.mobile_audio`, so they reach only Routes that enable the RabiLink/mobile endpoint and never merge into standalone `speech`. Replies continue through `/api/agent/replies` into the RabiLink downlink. Android does not retain one 24-hour raw recording; PC-finalized ASR segments and Agent TTS use per-file rolling 24-hour caches. Chunk sequences must increase continuously; after 15 seconds without PCM, the PC retires the stale stream and restores the previous audio input.
- Glasses mode keeps CXR and the native message bridge in the foreground service and starts the glasses app. The glasses app records continuously after setup; the default action is immediate push, and one touchpad tap prompts the Agent. TTS uses one ordered Classic-BT sequence, `PLAYBACK_BEGIN → PCM → PLAYBACK_END`. Glasses pause capture, validate message identity and byte count, return `played` only after their own `AudioTrack` marker is reached, then resume capture. A lost phone message service reconnects automatically with a 1.5–30 second exponential backoff while retaining the manual reconnect action.
- Both modes share route personas, durable text/control/media queues, cursor, chat history, downlink-TTS settings, and action gates. Enabling glasses does not create another account or conversation stack. ASR/VAD/segmentation/language settings belong only to target-PC RabiSpeech and are not duplicated on Android.
- Phone/glasses PCM follows the same host boundary as remote Rabi TTS/ASR clients: the remote endpoint submits only an audio stream, while the target PC processes it and stores `sourceHostId/sourceHostName`, opaque voiceprint IDs, and decision evidence in the host-wide message. The host decides neither who someone is nor who the user is. Every connected persona keeps its own `conversation/current.jsonl` and may independently maintain `voice/voice-identities.jsonl`.
- The phone remains the reliable conversation and downlink owner, while `sourceDeviceKind` freezes the physical origin of every queued input. Glasses microphone audio, photos, and touchpad review requests use `sourceDeviceKind=glasses`; phone audio uses `sourceDeviceKind=mobile`. Both keep the stable `sourceDeviceId` of the companion backend that subscribes to Relay downlink events, so an ordinary reply returns to that phone before the phone presents it on-screen or forwards it to the glasses. The current PCM connection is recorded separately as `sourceStreamId`; a stream ID carrying `-phone-audio` or `-glasses-audio` must never become the reply device. A shared `sessionId` still lets Agents and audits distinguish the physical source through `sourceDeviceKind/channelType` without treating a mode switch as a new conversation. `routeProfileId` selects the receiving persona/Route only; it never identifies the source as the role panel. A mobile-audio AgentPacket must retain `targetType=rabilink` and `adapterType=rabilink`.
- `RabiConversationService` is the single owner of input mode, with only `PAUSED`, `PHONE`, and `GLASSES`. Applying settings always stops the non-target capture path first: glasses mode pauses phone `AudioRecord`, phone mode closes the CXR/Phone SDK glasses bridge, and disabling continuous listening stops both paths. Two microphones therefore cannot continue uploading in parallel after a mode switch.

## Reliability and security

- Text/control uplink: at most 2,000 items, stable IDs, and automatic offline replay. Unconfirmed entries are no longer silently deleted by age; a full queue rejects new work with an explicit error. Continuous PCM does not enter this disk queue.
- Text and media freeze both `routeProfileId` and `clientMessageId` before entering a background queue. The UI reports queued, sending, handed to Rabi PC, or a concrete failure; later navigation cannot retarget queued work.
- Media uplink: at most 500 local items and 64 MiB each. Unconfirmed local entries are not age-pruned; a full queue explicitly rejects new media. Relay temporary media objects still use a seven-day server TTL, application isolation, and authenticated downloads.
- Downlink: durable opaque cursor, `deliveryId`-based delivery deduplication, and PCM cache. SSE `ready/outbox_available` only wakes the client, which then performs one cursor-bounded query to cover events missed offline. Relay sends a transport keepalive every 15 seconds; after 45 seconds without any SSE bytes, Android treats only the half-open socket as stalled and reconnects it. The restored connection still performs one `ready → cursor` catch-up and never polls business state. A normal Relay restart preserves the shared cursor generation; a runtime-state rollback or invalid cursor returns `cursorReset=true`. The phone then replays retained messages through its local terminal-state deduplication and stores the replacement cursor, so a client cursor cannot remain permanently ahead of restored server state. Relay does not TTL-delete a message with explicit `targetDeviceIds` until every explicit target has returned `delivered`; broadcasts and kind-only messages retain bounded TTL semantics. A TTS item yields the head after three failures and remains retryable.
- Receipts: `delivered` proves only that the mobile endpoint received/presented the message; it is never equivalent to `played`. A phone returns `played` only after its local `AudioTrack` marker, and glasses return it only after their own marker through Classic BT. `delivered/played/playback_failed` first enter the phone-private `receipt-queue`, which serves both crash-safe local deduplication and reliable replay after connectivity returns. Relay persists receipts and emits `outbox_receipt`; it never guesses device playback state.
- Glasses voice delivery is acknowledged only when the phone SDK is initialized, the device is authenticated, and both the Classic BT message and audio channels are online. If the channel is not ready, delivery fails into the existing deferred retry path instead of advancing the cursor merely because an SDK object exists.
- The phone and glasses APKs share `RabiGlassAudioProtocol` as the single source for commands, message prefixes, client ID, and the audio stream tag; neither packaged side keeps a duplicate protocol literal set.
- Attachment-only downlink needs no fabricated body text: images, video, audio, and arbitrary files are downloaded, recorded in chat, and announced with a normal notification even when text is empty.
- The production Android backend has removed the legacy whole-utterance ASR disk queue and the bypass that called remote `audio/transcriptions` before manually publishing a speech message. Until PC acknowledgement, Android retains the current pending PCM chunk, its sequence, and a stable `chunkId`. Same-stream sequence retries and cross-stream retries after a lost ACK are both deduplicated by RabiSpeech using `sourceDeviceId + chunkId + PCM SHA-256`, so the PCM is not fed into ASR twice. Android connectivity events and a restored RabiLink SSE connection trigger immediate continuation; only temporary service failures use one-shot backoff. The PCM executor and newest-audio buffer are bounded. Prolonged weak-network blocking retains the acknowledgement-sensitive chunk, discards obsolete PCM, and reconnects under a new transient `sourceStreamId`, catching up to the live stream instead of growing memory or remaining permanently behind; the stable reply target `sourceDeviceId` remains unchanged. A one-shot 15-second PC expiry retires the old inactive stream. Android process death discards unacknowledged in-memory PCM rather than presenting it as durably saved. Text, control, media, and downlink TTS keep their existing durable queues and explicit retry paths.
- Device diagnostics retain at most 500 redacted, coarse events for seven days, write identical events at most once per minute, and never include chat text, transcripts, tokens, or request bodies.
- Phone capture supervision: `RabiPhoneAudioCapture` exclusively owns `AudioRecord`, a partial wake lock, durable capture metrics, a 45-second stall detector, and bounded 1–30 second restart backoff. Stall detection schedules one deadline from the last successful read and cancels the previous generation on pause, mode switch, or recorder restart instead of running a fixed watchdog interval. `RabiConversationService` only coordinates transport, notifications, and phone/glasses mode. Chat shows session duration, latest audio age, cumulative PCM bytes, and automatic recovery count.
- Audio cache and records: ASR audio, transcripts, and voiceprint evidence are stored only by PC RabiSpeech under per-file 24-hour retention. Android no longer creates an ASR audio cache or ASR metadata. It stores PCM under `audio-cache/tts-audio/` only for downlink speech that must play on the phone or glasses, while `speech-records/` keeps safe relative paths, `audio_expires_at`, and append-only TTS metadata. Text, control, media, and failed-downlink queues remain separate from playback cache and persona history.
- Reboot recovery persists message-transport intent independently from continuous listening. A started text, media, or downlink connection restores its cursor, durable queues, and both notifications as a `dataSync` foreground service after process or device restart even when microphone capture is disabled; an explicit Stop disables future automatic restore. Where Android forbids microphone startup from a boot broadcast, opening the app resumes continuous capture without dropping queued messages.
- Tokens, chat, TTS, and attachments stay in app-private storage. PC-to-mobile files remain constrained by `allowedFileRoots`.

## Acceptance boundary

Android and TypeScript automation prove protocol, queue, receipt-state-machine, and build behavior only. Release acceptance still requires a real Android phone and Rokid glasses for long-running background capture, lock/unlock recovery, CXR reconnection, physical touchpad input, live battery refresh, actual phone/glasses speaker playback, notification permission, and vendor battery-management behavior.

Use `apps/rabilink-android/scripts/Test-RabiMobileAudioSoak.ps1` for the phone-side 24-hour acceptance run. It periodically reads the foreground-service state and app-private capture metrics, preserves JSONL evidence, and checks service residency, latest-audio age, and continued PCM-byte growth. Run it on every target release device class; an emulator or a successful build is not equivalent evidence.

On 2026-07-18, phone-side smoke validation passed on a physical Xiaomi Android device: in-place APK install, initialized default chat, expired-login recovery, foreground service, both ongoing notifications, and a real tap on “Prompt Rabi” producing a durable control-queue item. No app crash or foreground-service permission failure was observed. Physical Rokid-glasses validation is still outstanding, so this does not establish full release validation.

The 2026-07-22 code-side closeout added durable phone/glasses source attribution and the glasses audio-channel delivery gate. The mobile audit, Relay media regression, and both phone and glasses APK builds pass. This still does not replace physical Rokid verification for the touchpad, continuous capture, reconnection, and speaker playback.

`npm run check:rabilink:mobile` provides a stable repository audit for standalone chat, the phone backend, optional glasses, notifications, media, personas, speech, reboot recovery, retention of all 85 AIUI allowlisted configuration actions, and the Relay attachment regression.
